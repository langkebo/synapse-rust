# 数据库架构完整性与一致性全面审查 Spec

## Why
当前项目经历过多轮 Schema 迭代与修复，需要一次可追踪、可复验的全面数据库审查，确保数据库定义、迁移脚本、运行时查询与业务模型在结构、约束和性能层面保持一致，避免隐性数据风险在后续版本放大。

## What Changes
- 建立“数据库资产清单”，覆盖 `migrations/`、`schema.sql`、初始化脚本、运行时 SQL 与存储层模型
- 建立“结构一致性基线”，验证表/视图/索引/约束在定义与使用层面的匹配关系
- 建立“字段规范基线”，验证命名、类型、默认值、NULL 语义、索引策略与项目规则的一致性
- 建立“设计缺陷清单”，识别冗余、范式违背、潜在不一致点与审计字段缺口
- 产出“分级问题报告 + 重构优先级 + 优化建议 + 验证结果”
- 在开发环境落地高优先级修复，并执行完整性与性能对比验证

## Scope
- In Scope:
  - PostgreSQL 相关 Schema、迁移脚本、初始化与校验脚本
  - Rust 存储层与服务层中直接 SQL 查询及模型映射
  - 数据完整性测试、迁移回归测试、关键查询性能基准
- Out of Scope:
  - 第三方外部数据库服务的内部实现
  - 非数据库瓶颈导致的系统级性能问题

## Deliverables
- `docs/review/database-architecture-quality-report.md`
- `docs/review/database-refactor-priority.md`
- `docs/review/database-performance-benchmark.md`
- 必要的迁移脚本与代码修复（按任务分阶段提交）
- 验证结果记录（完整性、回归、基准对比）

## ADDED Requirements

### Requirement: 数据库资产全量扫描
系统 SHALL 建立覆盖数据库定义与使用方的全量清单，作为后续审查唯一基线。

#### Scenario: 扫描数据库资产
- **WHEN** 扫描项目数据库相关文件
- **THEN** 必须覆盖 `migrations/`、`schema.sql`、`docker/init`、`scripts/*migration*`
- **AND** 必须收集 Rust 代码中的 SQL 语句与模型定义
- **AND** 输出统一资产清单并标记来源

### Requirement: 架构完整性验证
系统 SHALL 验证数据库对象定义完整且可维护。

#### Scenario: 表与视图完整性检查
- **WHEN** 对比统一 Schema 与迁移链路
- **THEN** 所有业务关键表必须存在且可被最新迁移构建
- **AND** 运行时被引用对象必须在 Schema 中可解析
- **AND** 缺失对象必须进入严重问题清单

#### Scenario: 关系与约束完整性检查
- **WHEN** 检查实体关系
- **THEN** 主键、外键、唯一约束、检查约束应满足业务规则
- **AND** 外键引用方向与删除策略需与业务一致
- **AND** 约束缺失或失效必须提供修复 SQL 方案

### Requirement: 字段规范一致性验证
系统 SHALL 按项目规则验证字段命名、类型和约束语义。

#### Scenario: 命名规范检查
- **WHEN** 扫描字段命名
- **THEN** 布尔字段应遵循 `is_/has_` 前缀策略
- **AND** 时间字段应遵循 `*_ts` 或 `*_at` 规则
- **AND** 历史兼容字段需标记并给出收敛方案

#### Scenario: 类型与长度合理性检查
- **WHEN** 审核字段定义
- **THEN** PostgreSQL 类型与 Rust 类型映射应一致
- **AND** 不应无依据过度使用 `TEXT/BLOB`
- **AND** 长度/精度/可空性/默认值应符合业务语义

#### Scenario: 索引有效性检查
- **WHEN** 评估索引结构
- **THEN** 高频过滤与排序字段应具备可用索引
- **AND** 重复、低选择性、无使用价值索引应被识别
- **AND** 每条索引建议应包含收益与潜在写入开销说明

### Requirement: 设计缺陷识别与范式评估
系统 SHALL 识别影响一致性与可维护性的结构设计问题。

#### Scenario: 冗余与范式检查
- **WHEN** 评估核心业务表
- **THEN** 应识别重复字段与可拆分依赖
- **AND** 至少验证第三范式满足情况
- **AND** 违反范式的设计需给出渐进式重构路径

#### Scenario: 审计字段检查
- **WHEN** 检查业务关键表
- **THEN** 应具备必要审计字段（如 `created_ts`、`updated_ts`）
- **AND** 若使用其他命名需提供统一收敛策略

### Requirement: 问题分级与改进方案
系统 SHALL 输出可执行的问题治理方案。

#### Scenario: 分级报告生成
- **WHEN** 审查完成
- **THEN** 所有问题按严重/中等/轻微分级
- **AND** 每项问题包含证据、影响、修复建议与验证方式
- **AND** 给出数据库重构优先级与分阶段实施建议

### Requirement: 修改落地与验证
系统 SHALL 在开发环境实施高优先级修复并完成验证。

#### Scenario: 数据完整性验证
- **WHEN** 应用修复迁移与代码修改
- **THEN** 必须通过主外键一致性、唯一性、空值语义、级联行为验证
- **AND** 必须通过迁移前后关键查询结果一致性检查

#### Scenario: 性能基准对比验证
- **WHEN** 完成修复后执行基准
- **THEN** 必须对关键慢查询进行前后对比
- **AND** 记录 P50/P95、扫描行数、索引命中情况
- **AND** 回归超阈值项必须进入待优化清单

#### Scenario: 业务需求满足验证
- **WHEN** 完成完整性与性能测试
- **THEN** 与数据库相关核心业务流程应可正常通过
- **AND** 不得引入兼容性回归

## Risks
- 历史迁移脚本存在分叉，导致“理论正确但重放失败”
- 运行时 SQL 与统一 Schema 漂移，导致线上隐性失败
- 过度索引优化可能提升读性能但降低写性能
- 范式修复若一次性推进，可能引发较大兼容风险

## Success Criteria
- 高严重问题全部闭环并完成复验
- 中等问题具备可执行排期与责任归属
- 关键链路完整性测试全部通过
- 关键查询性能无显著回归，且有可追踪的优化收益
