# 后端全面优化实施蓝图 Spec

## Why
当前 `API-TEST-OPTIMIZATION-PLAN.md` 已暴露出功能缺口、Schema Drift、迁移脚本累积失序、冗余脚本难治理与交付门禁不足等系统性问题。需要基于现有测试阶段缺陷、结合 Synapse 的模块化架构与工程治理实践，形成一套可执行、可审计、可回滚的后端优化总方案。

## What Changes
- 建立覆盖功能完整性、数据库治理、迁移脚本治理、脚本清理、交付验收五大部分的统一优化方案
- 输出缺失核心业务能力清单，并为每项能力定义接口契约、状态机、异常码、性能目标与验收边界
- 设计 Schema Drift 根治机制，包括单一事实版本库、CI 漂移检测、三段式迁移与回滚演练
- 重构数据库迁移脚本治理规范，包括语义压缩、性能基线、命名规则与校验和策略
- 制定冗余脚本识别、保留、归档、删除与追溯闭环
- 定义交付物、测试门禁、预发布验证、签字放行与 PDF 产出要求

## Impact
- Affected specs: 后端功能规划、数据库变更治理、CI/CD 质量门禁、发布流程、审计与回滚能力
- Affected code: `src/web/routes/`, `src/services/`, `src/storage/`, `migrations/`, `scripts/`, `tests/`, `docs/`
- Affected systems: PostgreSQL、CI 流水线、预发布环境、监控告警、发布审批流程

## ADDED Requirements

### Requirement: 功能完整性补强治理
系统 SHALL 输出一份缺失核心业务功能全景清单，并按业务价值、上线阻塞性、测试风险和运维影响进行优先级分层。

#### Scenario: 识别缺失核心业务能力
- **WHEN** 对测试阶段缺陷、当前代码结构、现有接口实现与 Synapse 最佳实践进行交叉审计
- **THEN** 产出缺失能力清单，至少覆盖权限审计、批量操作、数据导出、监控告警、灰度开关等能力域
- **AND** 为每项能力标注当前状态、缺失原因、依赖模块、影响面与建议优先级

#### Scenario: 编写单项能力需求文档
- **WHEN** 某项缺失能力被纳入优化范围
- **THEN** 需求文档必须包含接口契约、请求响应结构、鉴权要求、状态机、异常码、幂等要求、性能指标与审计字段
- **AND** 文档需区分 Matrix 标准能力、Synapse 兼容能力与项目私有能力

#### Scenario: 制定迭代排期
- **WHEN** 所有能力需求被汇总
- **THEN** 将能力划分为 P0、P1、P2 三档
- **AND** 给出角色分工、人力投入、前置依赖、并行关系与阶段性交付目标

### Requirement: Schema Drift 根治机制
系统 SHALL 建立数据库结构单一事实版本库与自动化漂移检测机制，禁止绕过迁移体系直接改表。

#### Scenario: 建立单一事实版本库
- **WHEN** 设计数据库治理体系
- **THEN** 定义 baseline、migrate、undo 三段式脚本组织方式
- **AND** 规定所有表结构变更必须通过版本化脚本提交、评审与执行
- **AND** 明确人工改表为违规操作，并要求补录审计记录与纠正流程

#### Scenario: 引入漂移检测门禁
- **WHEN** 合并请求进入 CI 阶段
- **THEN** 自动对比代码分支 schema 与目标环境 schema
- **AND** 生成结构差异报告，至少覆盖表、字段、索引、约束、默认值与校验规则
- **AND** 若发现未备案漂移，流水线必须阻断合并

#### Scenario: 设计回滚与演练策略
- **WHEN** 任一迁移脚本进入发布链路
- **THEN** 必须提供幂等 down 脚本、适用前提、风险点与回滚验证步骤
- **AND** staging 必须完成回滚演练
- **AND** 回滚耗时目标不得超过 5 分钟

### Requirement: 迁移脚本优化与合并规范
系统 SHALL 对历史与新增迁移脚本建立统一的语义压缩、性能基线、命名与防篡改规范。

#### Scenario: 历史脚本语义压缩
- **WHEN** 对历史迁移脚本做存量治理
- **THEN** 合并连续对同一表字段的重复 ALTER 操作
- **AND** 减少不必要的表重建、锁表与数据回填次数
- **AND** 保留合并前后映射关系，确保可追溯

#### Scenario: 迁移性能评审
- **WHEN** 评审单条迁移脚本
- **THEN** 必须定义 1 千万行数据量下的执行性能基线
- **AND** 单条脚本执行时间目标应小于 30 秒
- **AND** 若超过阈值，则必须拆分执行、改写 SQL 策略或采用在线变更方案

#### Scenario: 统一命名与校验和
- **WHEN** 新增迁移脚本
- **THEN** 命名格式必须为 `V{版本}__{Jira编号}_{简短描述}.sql`
- **AND** 每条脚本必须附带 SHA-256 校验和与校验策略
- **AND** 校验失败时应阻断执行并要求人工复核

### Requirement: 冗余脚本生命周期治理
系统 SHALL 为数据库脚本建立生命周期标签、识别策略、保留窗口、归档机制与删除审计闭环。

#### Scenario: 标记冗余脚本
- **WHEN** 对现有脚本仓库执行盘点
- **THEN** 使用 `deprecated`、`unused`、`test-only` 标签描述脚本状态
- **AND** 通过静态扫描与运行时覆盖率双维度识别冗余脚本

#### Scenario: 执行脚本删除流程
- **WHEN** 某脚本被认定可删除
- **THEN** 先保留一个发布周期并标记为待删除
- **AND** 在观察期无异常后再物理删除
- **AND** 删除操作必须经过双人评审并记录到 `CHANGELOG-DB.md`

#### Scenario: 归档与紧急恢复
- **WHEN** 脚本被删除或合并
- **THEN** 必须保留 Git 归档备份与可追溯 tag
- **AND** 提供紧急恢复路径、触发条件与恢复验证步骤

### Requirement: 交付物与验证门禁
系统 SHALL 定义统一的优化方案交付物、测试指标、压测目标、预发布流程与上线准出机制。

#### Scenario: 输出正式交付物
- **WHEN** 优化方案定稿
- **THEN** 输出《后端优化实施方案》正式文档
- **AND** 文档内容必须覆盖详细设计、风险清单、回滚步骤、监控指标与实施路径
- **AND** 文档需支持导出为 PDF

#### Scenario: 定义自动化测试与性能门禁
- **WHEN** 为新增或整改能力制定验收标准
- **THEN** 单元测试覆盖率目标不得低于 80%
- **AND** 集成测试需覆盖所有新增接口
- **AND** 性能测试的 QPS 与 P99 目标不得低于现有峰值的 120%

#### Scenario: 执行预发布准出
- **WHEN** 功能与数据库治理项进入发布前验证
- **THEN** 预发布环境必须完成全量回归、压测、漂移检查与回滚演练
- **AND** 最终需生成由 QA、DBA、SRE 三方签字的上线准出报告
- **AND** 未满足任一门禁时不得进入生产发布

## MODIFIED Requirements

### Requirement: API 测试优化方案升级为后端治理总方案
现有以测试脚本治理为主的优化文档需要扩展为覆盖功能规划、数据库治理、迁移策略、清理流程与发布门禁的总方案。测试优化不再只是脚本修正，而是作为后端架构与交付治理的输入来源之一。

## REMOVED Requirements

### Requirement: 仅以脚本误配修复作为主要交付目标
**Reason**: 仅修复测试脚本无法根治当前暴露的功能缺失、Schema Drift 与迁移治理问题。
**Migration**: 将脚本治理内容收编为“验证与交付”章节，作为总方案中的一个执行工作流，而非唯一目标。
