# Synapse 对标差距分析与系统优化方案 Spec

## Why
当前 `synapse-rust` 已具备较广的 Matrix/Synapse API 覆盖面，但仓库内同时存在“开发中、部分实现”“已完成、生产就绪”“存在未完成与已知限制”等互相冲突的状态描述，导致项目真实成熟度、兼容性边界与发布风险难以被准确判断。对照 Matrix 规范、Element Synapse 的项目形态与最新发布节奏，当前项目的主要问题已从“有没有代码”转向“是否具备规范级证据、工程级治理与可验证的交付闭环”。

本 Spec 用于先输出一份面向项目负责人的系统性分析与详细优化方案，形成统一事实源，指导后续代码整改、测试补强、文档收敛与发布治理，而不是继续追加零散的审计报告。

## What Changes
- 建立“Matrix 规范 / Synapse 上游 / 本仓库实现”三层对标模型，输出可审计的兼容性基线
- 建立覆盖架构、兼容性、测试、文档、部署、运维、安全的系统性问题清单，并按 P0/P1/P2/P3 分级
- 建立单一事实源文档体系，统一 README、完成度报告、缺失功能、测试报告与发布结论
- 制定分阶段优化路线图，明确短期止血、中期收敛、长期演进三类任务与验收门槛
- 将“已实现”升级为“已被何种证据证明”，引入规范级互操作验证、联邦验证与可选能力分层治理
- 将后续整改映射到 `src/`、`tests/`、`docs/`、`.github/workflows/`、`scripts/`、`migrations/` 的执行任务

## Impact
- Affected specs: Matrix Client-Server / Server-Server / Admin 兼容基线、测试治理基线、文档治理基线、发布准入基线
- Affected code: `src/web/routes/`、`src/services/`、`src/storage/`、`src/worker/`、`src/bin/`
- Affected docs: `README.md`、`docs/synapse-rust/*`、`docs/*`、`.trae/specs/*`
- Affected validation: 单元测试、集成测试、联邦测试、E2E、性能测试、发布门禁
- **BREAKING**: 后续会把“完成度结论”从主观描述改为证据驱动，可能推翻当前部分“已完成/可发布”的口径

## ADDED Requirements

### Requirement: 建立统一对标基线
系统 SHALL 建立以 Matrix 规范、Element Synapse 上游能力和本仓库实现现状为核心的统一对标基线，并将每项能力映射到代码、测试、文档与状态结论。

#### Scenario: 三层基线建模
- **WHEN** 启动本轮系统分析
- **THEN** 必须区分以下三层对象：
  - Matrix 标准能力：Client-Server、Server-Server、Application Service、Identity、Push、Room Versions、Olm/Megolm、相关稳定 MSC
  - Synapse 上游能力：部署模型、Worker 形态、Admin API、联邦行为、SSO/OIDC/SAML、媒体与后台任务治理
  - 本仓库实现：路由、服务、存储、迁移、测试、CI、文档
- **AND** 每项能力都必须标注状态：`已实现并验证`、`已实现待验证`、`部分实现`、`未实现`、`不纳入本期`

#### Scenario: 证据绑定
- **WHEN** 输出任一兼容性结论
- **THEN** 必须同时绑定代码位置、测试或脚本、文档来源与风险说明
- **AND** 不允许仅以“已有路由”“已有文件”“报告曾声称已完成”作为完成依据

### Requirement: 输出系统性问题清单
系统 SHALL 建立覆盖架构、兼容性、测试、文档、部署、运维、安全与发布治理的问题清单，并按优先级分级管理。

#### Scenario: 问题分类
- **WHEN** 盘点仓库现状
- **THEN** 问题至少要分为以下类别：
  - 架构边界问题：大容器、过重路由聚合、单体膨胀、worker 边界不清
  - 兼容性问题：联邦、E2EE、SSO、AppService、Admin、可选 MSC 能力收敛不足
  - 测试问题：测试未接线、弱断言、跳过掩盖风险、缺少规范级互操作验证
  - 文档问题：README、完成度报告、缺失功能、测试报告互相冲突
  - 发布问题：迁移来源不唯一、运行时初始化与文档口径冲突、缺少统一准出基线

#### Scenario: 优先级分级
- **WHEN** 为问题分级
- **THEN** 必须采用 `P0/P1/P2/P3`
- **AND** 至少满足以下含义：
  - `P0`：影响真实发布判断、数据一致性、联邦安全、关键兼容结论可信度
  - `P1`：影响核心链路稳定性或主要能力收敛，但存在替代路径
  - `P2`：影响维护效率、可观测性、性能与扩展能力
  - `P3`：体验优化、结构优化、长期演进项

### Requirement: 建立单一事实源文档体系
系统 SHALL 建立统一的状态事实源，所有“完成度”“兼容性”“可发布性”结论必须从同一套基线与证据自动或半自动生成。

#### Scenario: 文档收敛
- **WHEN** 收敛现有文档
- **THEN** 必须将 README、缺失功能、完成度报告、测试分析、优化计划纳入统一口径
- **AND** 必须明确以下信息：
  - 当前版本支持哪些 Matrix 标准能力
  - 哪些能力仅对齐 Synapse 行为但尚未做互操作验证
  - 哪些功能属于项目私有扩展，不得冒充标准兼容能力
  - 哪些功能为可选能力，需要外部依赖或企业场景验证

#### Scenario: 结论生成规则
- **WHEN** 任一文档声称“已完成”“生产可用”“兼容 Synapse”
- **THEN** 必须附带版本范围、证据范围、未覆盖边界、剩余风险与验收标准
- **AND** 与事实源冲突的旧文档必须被修订、归档或显式标记为历史材料

### Requirement: 建立规范级验证体系
系统 SHALL 将现有项目内测试升级为“项目回归验证 + 规范级互操作验证”双层结构，以证明兼容性而非仅证明代码可运行。

#### Scenario: 测试分层治理
- **WHEN** 定义验证体系
- **THEN** 必须至少包含：
  - 单元测试：核心业务状态机、错误处理、数据转换
  - 集成测试：路由、鉴权、数据库、副作用
  - 联邦测试：跨 homeserver 行为、签名、鉴权、房间同步
  - 互操作测试：对齐 Matrix 生态常用规范测试资产
  - 性能与稳定性测试：关键路径、迁移、长连接、后台任务
- **AND** 所有测试层都必须明确执行入口、CI 接线方式与产物位置

#### Scenario: “已实现”到“已验证”的升级
- **WHEN** 标记某功能为兼容或完成
- **THEN** 必须说明该结论是由哪类测试证明
- **AND** 如仅通过本地脚本或 smoke test，不得升级为规范级兼容结论

### Requirement: 制定详细优化路线图
系统 SHALL 输出按阶段推进的详细优化方案，覆盖止血、收敛、增强、治理四个层次，并定义每阶段的交付物、依赖关系与退出条件。

#### Scenario: 路线图阶段划分
- **WHEN** 制定路线图
- **THEN** 至少划分以下阶段：
  - `M0 基线清点`：统一事实源、完成问题分级、冻结当前口径
  - `M1 P0 止血`：修复文档冲突、补齐测试接线、梳理发布与迁移真实入口
  - `M2 核心能力收敛`：联邦、E2EE、Admin、SSO、AppService、Worker 等核心域逐项收敛
  - `M3 质量门禁`：互操作测试、性能基线、发布门禁、文档质量门禁落地
  - `M4 架构演进`：拆分过重模块、沉淀长期治理与运维标准

#### Scenario: 每阶段交付物
- **WHEN** 定义每个里程碑
- **THEN** 必须包含：目标、范围、输入、输出、负责人角色、验收标准、回归要求、风险与回滚策略

### Requirement: 明确重点整改方向
系统 SHALL 将优化方案拆分为可执行的专题主线，而不是仅输出泛化建议。

#### Scenario: 专题主线
- **WHEN** 输出详细优化方案
- **THEN** 至少包含以下主线：
  - 兼容性主线：按 Matrix 标准域与 Synapse 关键能力域构建差距矩阵
  - 架构主线：收敛 `ServiceContainer`、总路由装配、worker 与主服务边界
  - 测试主线：补齐测试接线、去除弱断言、引入互操作验证
  - 文档主线：建立唯一事实源与自动更新机制
  - 发布主线：统一迁移入口、运行手册、发布准出与回滚基线

#### Scenario: 重点专项
- **WHEN** 落地重点专项
- **THEN** 优先处理以下高风险主题：
  - README 与运行时行为冲突
  - 测试文件存在但未纳入执行入口
  - 文档“全部完成”与“存在已知限制”并存
  - 联邦/Worker/SSO 等对标 Synapse 关键能力缺乏验证
  - `已实现` 与 `可发布` 结论缺乏证据链

### Requirement: 定义最终输出物
系统 SHALL 形成既能指导内部执行、又能对外说明项目成熟度的标准化输出物集合。

#### Scenario: 输出物最小集合
- **WHEN** 本轮分析完成
- **THEN** 至少要形成以下输出物：
  - 差距分析报告
  - 能力兼容矩阵
  - P0/P1/P2/P3 问题台账
  - 分阶段优化路线图
  - 验证体系与门禁方案
  - 文档收敛与发布治理方案

## MODIFIED Requirements

### Requirement: 完成度文档从“结论型”升级为“证据型”
现有完成度、测试结论与可发布性文档 SHALL 从人工叙述升级为基于统一基线、统一状态枚举和统一证据来源的输出物。

## REMOVED Requirements
无
