# Task 14 - 搜索链路统一与性能优化执行文档

## 1. 目标

统一当前搜索实现路径，定义查询 DSL、provider 抽象和底层执行策略，逐步淘汰长期保留的低效旁路查询，并建立可量化的性能基线。

## 2. 范围

- 搜索入口：全局搜索、房间内搜索、相关兼容路径
- 搜索能力：查询参数、分页、排序、过滤、高亮、索引初始化
- 底层实现：Postgres FTS 为最小落地，预留外部 provider 扩展位
- 非目标：本任务不直接切换到外部搜索后端

## 3. 输入

- `tasks.md` 中 `Task 14 详细说明`
- `spec.md` 中“统一搜索实现路径”要求
- 当前搜索路由、服务和存储层实现
- 与 schema、索引相关的历史材料

## 4. 输出

- 搜索链路架构方案
- 查询 DSL 与 provider 抽象设计
- 性能基线、索引策略与迁移计划

## 5. 执行阶段

### Phase 1: 现状盘点

- 盘点所有搜索入口和旁路 SQL
- 记录分页、排序、过滤、高亮行为差异
- 标记索引依赖和潜在性能热点

### Phase 2: DSL 与 provider 设计

- 定义统一查询 DSL
- 定义 provider 接口与能力边界
- 规定请求层、领域层、执行层之间的数据流

### Phase 3: 性能治理方案

- 设计 Postgres FTS 最小实现方案
- 识别需要淘汰的低效 `LIKE` 兜底路径
- 定义索引策略、字段规范化和排序一致性规则

### Phase 4: 迁移计划

- 设计从现状到统一链路的阶段性迁移路径
- 定义兼容策略、降级方案和回滚条件
- 规定基准验证方法和目标

## 6. 技术约束

- 必须兼容现有 Matrix/Synapse 搜索语义
- DSL 不得泄漏底层实现细节
- provider 需要支持多后端扩展
- 性能策略必须可通过基准或查询计划验证

## 7. 里程碑与时限

- D1-D2：完成现状盘点和问题分类
- D3-D4：完成 DSL、provider 和索引策略设计
- D5：完成迁移计划和性能基线定稿

## 8. 质量指标

- 搜索入口覆盖率达到 100%
- 所有长期低效旁路查询都被标记或纳入淘汰计划
- 至少形成 1 组可量化性能目标
- 兼容层、领域层和执行层边界清晰

## 9. 测试与验证

- 语义验证：为 DSL 定义等价查询样例
- 性能验证：定义基准测试或 `EXPLAIN ANALYZE` 样例
- 兼容验证：覆盖分页、排序、过滤与高亮
- 实施验证：验证 provider 切换或降级场景

## 10. 风险与缓解

- 风险：统一 DSL 后与现有请求行为不兼容
- 缓解：保留兼容层映射，先统一内部模型

- 风险：性能方案依赖索引调整，落地成本高
- 缓解：先定义 Postgres FTS 最小闭环，再扩展增强

- 风险：多 provider 设计过早复杂化
- 缓解：只保留最小扩展接口，不提前实现全部能力

## 11. 验收标准

- 已形成统一搜索链路架构图与接口定义
- 已明确 Postgres FTS 最小落地方案
- 已形成性能基线和迁移顺序
- 已建立可执行的测试与基准验证要求
