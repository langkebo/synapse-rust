# 文档治理与单一事实源规则（2026-04-05）

> 文档定位：治理规范
> 目标：明确当前仓库哪些文档可以表达“正式状态”，哪些只能表达“阶段记录”

---

## 1. 单一事实源

当前正式能力状态的单一事实源为：

- `docs/synapse-rust/CAPABILITY_STATUS_BASELINE_2026-04-02.md`

使用规则：

1. 当前能力状态、成熟度、验证结论，以该文档为准；
2. 其他文档若涉及能力描述，必须回指该基线；
3. 没有测试证据、CI 证据或明确代码闭环的能力，不得在其他文档中升级表述。

---

## 2. 文档分层

### A. 权威文档

可承载当前正式判断：

- `CAPABILITY_STATUS_BASELINE_2026-04-02.md`
- 本轮新增治理文档（执行计划、测试语义、false green 治理等）

要求：
- 结论必须有证据；
- 结论必须可维护；
- 结论必须与当前代码和 CI 行为一致。

### B. 派生文档

可做说明、索引、导航，但不独立下当前成熟度结论：

- `README.md`
- `TESTING.md`
- API 说明文档
- 部署与操作指南

要求：
- 只能复述或引用权威结论；
- 必须标明适用范围；
- 不得把历史结果表述成当前承诺。

### C. 归档/历史文档

仅保留阶段记录，不作为当前发布口径：

- `COMPLETION_REPORT.md`
- 各类阶段性总结、审计快照、旧版分析报告

要求：
- 必须标注“历史材料 / 归档报告”；
- 必须显式回指当前权威基线；
- 不得再给出未限定语境的“生产就绪”“100% 完整实现”等当前判断。

---

## 3. 更新触发条件

以下情况必须同步检查文档：

1. CI 主门禁发生变化；
2. 测试执行语义发生变化（例如 ignored、环境变量门控、失败策略变化）；
3. capability baseline 中某个能力域状态变化；
4. placeholder 路由被实现、明确不支持，或纳入 allowlist；
5. 对 README 的能力概览做出影响用户判断的修改。

---

## 4. 文档写作约束

- 路由存在不等于能力完成；
- 测试文件存在不等于测试真实执行；
- 非阻断 evidence 不等于发布门禁；
- 历史通过记录不等于当前持续有效；
- “已实现并验证”必须能回溯到明确证据。

推荐表述：

- “已接线 / 已存在实现”
- “已纳入主门禁”
- “已补充契约测试”
- “当前仍需进一步验证”

避免表述：

- “100% 完整实现”
- “生产就绪”
- “全部通过，因此可对外承诺”

除非已被当前权威基线明确支持。

---

## 5. 与评审流程的关系

代码评审和文档评审至少检查：

1. 是否新增了高于证据强度的能力表述；
2. 是否把非阻断结果误写成阻断门禁；
3. 是否把 ignored / skip / 手动验证误写成自动通过；
4. 是否忘记回指基线文档。

可与 `docs/synapse-rust/CODE_REVIEW_CHECKLIST.md` 联动使用。

---

## 6. 本轮结论

从 2026-04-05 起：

- 基线文档负责“当前状态”；
- README 负责“高层入口”；
- TESTING 负责“执行语义”；
- COMPLETION_REPORT 仅保留“历史记录”。

任何与上述分层冲突的旧表述，都应视为待清理漂移。
