# synapse-rust 周度执行计划

> 日期：2026-04-03  
> 目标：将下一步执行清单拆成可直接推进的一周内动作

## 一、周目标

本周只做三件事：

1. 彻底统一事实源引用
2. 收敛旧完成度报告的残留口径
3. 给联邦与 E2EE 补出最小验证清单

## 二、执行项

### Day 1：统一引用

- 检查 README、审查索引、整改清单、能力基线、执行摘要、下一步执行清单的互相引用。
- 统一所有“当前状态”入口，只保留能力基线作为正式状态口径。
- 验收：没有任何新文档在正文中引用旧完成度报告作为当前依据。

### Day 2：收敛历史材料

- 清点仍提到 `COMPLETION_REPORT.md`、`API_TEST_REPORT.md` 的文档。
- 将这些引用改写为“历史材料 / 仅供追溯 / 不作为当前事实源”。
- 验收：旧报告只保留历史定位，不再承担当前结论。

### Day 3：固定测试门禁

- 在测试相关文档中明确主门禁、扩展验证、手动分析三类入口。
- 说明哪些测试影响发布，哪些仅作参考。
- 验收：任何测试条目都能明确归类，不再混用。

### Day 4：补联邦最小闭环

- 为联邦列出最小子能力清单：路由、签名、发送、接收、互操作。
- 每项写清代码证据、测试证据、文档来源、剩余风险。
- 验收：联邦不再只有笼统结论。

### Day 5：补 E2EE 最小闭环

- 为 E2EE 列出最小子能力清单：设备、密钥、备份、交叉签名、恢复。
- 每项写清状态，不再使用“完整实现”类总括说法。
- 验收：E2EE 状态可以按子能力阅读。

### Day 6：明确 AppService / Worker

- 写明 AppService 当前可承诺范围与未成熟边界。
- 写明 Worker 当前是单进程可用还是多进程未成熟。
- 验收：对外口径与能力基线一致。

### Day 7：复核与归档

- 检查本周新增或更新文档是否都落在正确分类：正式事实源 / 执行清单 / 历史归档。
- 更新索引页，确保新成员能快速找到应读文档。
- 验收：本周产物可独立被阅读和复核。

## 三、完成提示

当以下条件都满足时，说明这一周的文档优化推进完成：

- 引用关系统一。
- 历史报告降级完成。
- 测试门禁边界清晰。
- 联邦与 E2EE 有最小验证清单。
- AppService 与 Worker 的定位已明确。

## 四、下一步建议

如果继续推进，优先顺序如下：

1. 统一事实源引用
2. 收敛历史材料
3. 固定测试门禁
4. 补联邦 / E2EE 最小闭环
5. 明确 AppService / Worker 定位
