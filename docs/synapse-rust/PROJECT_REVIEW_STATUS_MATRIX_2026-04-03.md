# synapse-rust 状态矩阵

> 日期：2026-04-03  
> 目的：说明“正式事实源 / 执行清单 / 周度计划 / 历史归档”之间的关系，避免不同类型文档互相承担错误职责。

## 一、状态分层

| 层级 | 文档类型 | 代表文档 | 当前角色 | 允许承担的内容 | 不应承担的内容 |
|------|----------|----------|----------|----------------|----------------|
| L1 | 正式事实源 | `CAPABILITY_STATUS_BASELINE_2026-04-02.md` | 当前能力状态总表 | 能力状态、证据链、正式口径 | 行动排期、周计划、对外泛化结论 |
| L2 | 导航入口 | `PROJECT_REVIEW_INDEX_2026-04-03.md` | 读文档的入口 | 分类、推荐阅读顺序、文档定位 | 具体结论、能力宣称 |
| L3 | 执行汇总 | `PROJECT_REVIEW_EXECUTIVE_SUMMARY_2026-04-03.md` | 对外简报摘要 | 一句话结论、主要差距、下一步建议 | 作为唯一事实源下结论 |
| L4 | 执行清单 | `PROJECT_REVIEW_NEXT_ACTIONS_2026-04-03.md` | 短周期推进项 | 优先事项、验收标准、推进顺序 | 替代能力基线、单独承担完成度结论 |
| L5 | 周度计划 | `PROJECT_REVIEW_WEEKLY_PLAN_2026-04-03.md` | 日级动作拆分 | Day 1~Day 7 的推进安排 | 作为正式状态声明 |
| L6 | 长期分析 / 回溯 | `PROJECT_REVIEW_AND_OPTIMIZATION_PLAN_2026-04-03.md`、`PROJECT_REVIEW_ACTION_BACKLOG_2026-04-03.md` | 深入分析与整改台账 | 问题分析、路线图、P0/P1/P2 项 | 覆盖正式事实源或单独替代基线 |
| L7 | 历史归档 | `COMPLETION_REPORT.md`、`API_TEST_REPORT.md`、`FEATURE_IMPROVEMENT_REPORT.md` | 仅用于追溯 | 过去阶段记录、历史背景 | 当前状态判断、发布依据、对外口径 |

## 二、阅读关系

1. 先看 `CAPABILITY_STATUS_BASELINE_2026-04-02.md`，确认当前正式事实源。
2. 再看 `PROJECT_REVIEW_INDEX_2026-04-03.md`，按入口定位需要读的文档。
3. 需要对外汇报时，优先看 `PROJECT_REVIEW_EXECUTIVE_SUMMARY_2026-04-03.md`。
4. 需要推进工作时，先看 `PROJECT_REVIEW_NEXT_ACTIONS_2026-04-03.md`，再看 `PROJECT_REVIEW_WEEKLY_PLAN_2026-04-03.md`。
5. 需要追溯旧结论时，才回看历史归档材料。

## 三、口径规则

- 正式事实源只能有一套。
- 执行类文档可以拆分，但不能反向替代事实源。
- 历史材料可以保留，但必须显式降级为追溯用途。
- “已完成 / 生产就绪 / 完整实现”类措辞，只能出现在有证据边界的场景，不能覆盖全局状态。
- 任一文档如果开始承担不属于自己的职责，应优先回链到 L1。

## 四、推荐使用方式

- 需要判断“现在是什么状态”时，看 L1。
- 需要判断“先做什么”时，看 L4 和 L5。
- 需要判断“怎么对外说”时，看 L3。
- 需要判断“为什么这么安排”时，看 L6。
- 需要判断“以前发生过什么”时，看 L7。
