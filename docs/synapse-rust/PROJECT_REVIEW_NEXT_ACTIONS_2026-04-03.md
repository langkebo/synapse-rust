# synapse-rust 下一步执行清单

> 日期：2026-04-03  
> 目的：把当前审查结论拆成更短周期、可直接推进的动作

## 一、执行目标

本清单用于承接 `PROJECT_REVIEW_EXECUTIVE_SUMMARY_2026-04-03.md` 的下一步建议，把工作从“文档口径收口”推进到“事实源统一、证据补齐、能力分级稳定”。

## 二、执行原则

1. 先事实源，后派生文档。
2. 先 P0 收口，后 P1 补证，最后 P2 结构化。
3. 所有“已完成”都要能回链到代码、测试、文档证据。
4. 任何专题报告都不得单独承担全局完成度结论。

## 三、短期优先事项

### 1. 继续统一事实源

- 目标：确保项目状态、能力结论、测试说明都回到同一套基线。
- 动作：检查 README、审查索引、整改清单和能力基线的引用是否一致。
- 验收：不再出现多个版本的“当前状态”口径。

### 2. 继续收敛历史材料

- 目标：把旧完成度报告完全降级为追溯材料。
- 动作：检查仍在引用 `COMPLETION_REPORT.md`、`API_TEST_REPORT.md` 的文档。
- 验收：正式文档不再依赖旧完成度报告结论。

### 3. 明确测试门禁边界

- 目标：把主门禁、扩展验证、手动分析分开。
- 动作：在测试文档中固定每类测试的用途、入口和是否阻断发布。
- 验收：任何测试说明都能明确判断是否影响发布。

### 4. 补齐联邦与 E2EE 最小闭环证据

- 目标：让最关键的两个能力域先具备最小验证闭环。
- 动作：为联邦、E2EE 分别列出最小子能力和验证点。
- 验收：能力状态不再停留在笼统描述。

### 5. 明确 AppService 与 Worker 当前定位

- 目标：避免把“存在”误读成“成熟可用”。
- 动作：分别给出可承诺范围、未成熟边界和后续验证要求。
- 验收：对外口径与能力基线一致。

## 四、推荐推进顺序

1. 统一事实源
2. 收敛历史材料
3. 固定测试门禁
4. 补联邦与 E2EE 证据
5. 明确 AppService / Worker 定位

## 五、阶段完成提示

当以下条件满足时，可认为本阶段推进完成：

- 只存在一套正式事实源。
- 旧完成度报告均已降级为历史材料。
- 测试门禁说明清晰可执行。
- 联邦与 E2EE 已有最小闭环验证点。
- AppService 与 Worker 的当前定位已稳定。

## 六、建议下一步

如果继续推进，优先从以下两项开始：

- 统一事实源与文档引用
- 补齐联邦 / E2EE 的验证清单
