# synapse-rust 项目整改行动清单

> 来源：`PROJECT_REVIEW_AND_OPTIMIZATION_PLAN_2026-04-03.md`  
> 目标：把审查结论转成可以直接执行的整改任务  
> 范围：文档治理、测试门禁、迁移口径、能力分级、架构收口

---

## 一、执行原则

1. 先统一事实源，再处理派生文档。
2. 先修 P0，再修 P1，最后做 P2。
3. 所有能力结论必须有代码、测试、文档三类证据中的至少两类支撑。
4. 不再新增“完成度结论”，只做“验证后升级”。

---

## 二、P0：先止血

### P0-1 统一项目状态事实源

- 目标：只保留一个正式能力状态总表。
- 产物：`CAPABILITY_STATUS_BASELINE_2026-04-02.md` 的后续版本或同类总表。
- 验收：README、能力总表、发布说明、测试说明对同一能力的描述一致。

### P0-2 收敛 README 对外口径

- 目标：README 只保留定位、安装、运行、迁移、测试入口。
- 产物：README 更新稿。
- 验收：不再出现与历史报告冲突的“生产就绪”“100% 完整”等表述。

### P0-3 统一迁移入口说明

- 目标：把数据库初始化/迁移口径统一到唯一入口。
- 产物：README + 部署/测试文档统一说明。
- 验收：部署、CI、本地运行三处口径一致。

### P0-4 明确测试门禁边界

- 目标：区分主门禁、扩展验证、手动分析三类测试。
- 产物：测试策略说明或测试总览表。
- 验收：每个测试入口都能被明确归类，且是否阻断发布清晰。

### P0-5 处理过度结论文档

- 目标：把旧的强结论文档改为归档材料，不再作为当前状态依据。
- 产物：历史文档更新或归档标识。
- 验收：当前正式文档不再与归档文档冲突。

---

## 三、P1：补证据

### P1-1 建立联邦能力矩阵

- 目标：把联邦端点、行为、边界、剩余风险逐项列清。
- 产物：联邦能力矩阵文档。
- 验收：每个联邦子能力都能定位到代码和测试证据。

### P1-2 建立 E2EE 能力矩阵

- 目标：区分设备、密钥、备份、交叉签名、群组加密等子能力。
- 产物：E2EE 能力矩阵文档。
- 验收：不再使用“完整实现”这类笼统表述。

### P1-3 明确 Worker 当前定位

- 目标：说明单进程可用、多进程未成熟，避免误解。
- 产物：Worker 定位说明。
- 验收：公开口径和能力基线一致。

### P1-4 明确 SSO / OIDC / SAML 为可选能力

- 目标：将企业认证类能力从主承诺中分离。
- 产物：能力分层说明。
- 验收：主通过率不与可选能力混算。

### P1-5 建立最小互操作验证清单

- 目标：为核心能力补最小闭环验证。
- 产物：验证用例清单。
- 验收：联邦、E2EE、Admin、AppService 至少各有最小闭环验证点。

---

## 四、P2：收口结构

### P2-1 拆分总容器职责

- 目标：避免 `ServiceContainer` 继续膨胀。
- 产物：领域拆分方案。
- 验收：核心依赖按能力域分组，不再集中在单一大容器。

### P2-2 拆分总路由装配职责

- 目标：减少总路由中心堆叠。
- 产物：路由分层方案。
- 验收：标准路由、Admin、可选能力路由边界清晰。

### P2-3 建立文档索引与归档规则

- 目标：让新成员能快速知道该读哪份文档。
- 产物：文档索引页。
- 验收：正式文档、说明文档、历史材料可被明确区分。

### P2-4 建立性能与回滚基线

- 目标：把性能目标变成可验证基线。
- 产物：性能基线与运维基线说明。
- 验收：核心路径有基准值、退化阈值和回滚指引。

---

## 五、推荐执行顺序

1. P0-1 / P0-2 / P0-3
2. P0-4 / P0-5
3. P1-1 / P1-2 / P1-5
4. P1-3 / P1-4
5. P2-1 / P2-2 / P2-3 / P2-4

---

## 六、当前建议

如果要继续推进，下一步最值得做的是：

- 先把文档索引和事实源统一
- 再把测试入口和门禁说清楚
- 然后再做联邦和 E2EE 的证据补齐
