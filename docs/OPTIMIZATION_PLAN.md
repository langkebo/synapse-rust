## 一、文档定位

本方案用于替换现有偏“问题罗列式”的优化计划，形成可用于评审、排期、交付与 PDF 归档的正式《后端优化实施方案》章节草案。方案遵循 Matrix/Synapse 风格：以接口契约为边界、以服务分层为实施单元、以迁移闭环和回归证据为发布准入。

### 1.1 目标定义

| 目标 | 说明 | 结果要求 |
|------|------|----------|
| 功能完整性补强 | 修正测试脚本与真实实现偏差，补齐未收敛能力 | 核心链路按契约稳定通过 |
| Schema Drift 根治 | 消除“代码已依赖、Schema 未闭环”问题 | 代码、迁移、Schema、门禁统一 |
| 迁移脚本优化合并 | 形成统一基线与增量治理节奏 | 新环境、升级环境均可重复执行 |
| 冗余脚本清理 | 下线重复、过时、已被吸收的历史脚本 | 迁移目录可审计、可维护 |
| 交付验证标准化 | 将验收标准前移到测试与 CI | 发布前具备完整证据链 |

### 1.2 设计原则

| 原则 | 要求 |
|------|------|
| 契约优先 | 先定义路径、权限、状态码、错误码、响应字段，再落实现有 handler 与 service 收敛 |
| 分层收口 | `web/routes` 负责协议与鉴权，`services` 负责业务编排，`storage` 负责 SQL 与事务边界 |
| Schema 单一真实来源 | 统一 schema、增量迁移、回滚脚本、门禁脚本必须同批更新 |
| Synapse 风格兼容 | 核心 Matrix Client API 与 Synapse Admin API 分层治理，不混用私有扩展与标准路径 |
| 证据可追溯 | 测试、迁移、校验、回滚、监控都必须有产物或记录可审计 |

## 二、当前基线与问题判断

### 2.1 仓库实施基线

| 维度 | 当前实情 | 判断 |
|------|----------|------|
| 分层结构 | 已形成 `web/routes`、`services`、`storage` 三层主结构 | 架构方向正确，但需继续以契约约束边界 |
| 数据库迁移 | 根目录同时存在 `00000000_unified_schema_v6.sql`、多个日期迁移、`99999999_unified_incremental_migration.sql`，并已建立 `incremental/`、`rollback/`、`hotfix/`、`archive/` | 治理框架已出现，但入口仍双轨并存 |
| 测试分层 | 已存在 `tests/unit`、`tests/integration`、`tests/e2e`、`tests/performance` | 分层完整，需从“存在测试”升级到“契约驱动测试” |
| 主 CI | `ci.yml` 已执行 repo-sanity、迁移、fmt、clippy、test、build、coverage | 主链可用，覆盖范围尚需聚焦真实发布风险 |
| DB 门禁 | `db-migration-gate.yml` 已接入 schema coverage、contract coverage、layout audit、pg_amcheck、logical checksum | 已具备根治 Schema Drift 的基础能力 |
| 文档门禁 | `docs-quality-gate.yml` 当前仅校验 `docs/db/DIAGNOSIS_REPORT.md` | 文档治理尚未覆盖顶层实施方案 |
| Drift 检测 | `drift-detection.yml` 已存在，但仍保留与当前迁移命名体系不完全一致的旧规则 | 需对齐当前迁移治理规范，避免“门禁存在但不命中实际对象” |

### 2.2 主要问题归类

| 类别 | 现象 | 根因 | 严重度 |
|------|------|------|--------|
| 功能契约不一致 | API 测试脚本与真实路由、方法、鉴权、返回结构存在偏差 | 规范、实现、回归脚本未同步维护 | P0 |
| Schema 闭环缺失 | 历史上存在代码已读写、迁移未建表或索引不齐 | 统一 schema、增量迁移、exceptions 清理节奏不同步 | P0 |
| 迁移资产膨胀 | 根目录历史脚本、统一增量脚本、目录治理并存 | 迁移整合完成一半，归档口径未完全统一 | P1 |
| 回归标准偏宽 | 大量用例只做关键字探测，未强校验 HTTP 状态码与 Matrix `errcode` | 测试目标更像探测脚本，而非正式验收套件 | P1 |
| 文档与 CI 脱节 | 实施文档、门禁范围、发布准入没有形成一体化说明 | 文档尚未升级为工程交付物 | P2 |

### 2.3 本轮优化边界

- 纳入本方案：核心 Matrix Client API、Synapse Admin API、Schema/迁移治理、测试分层、CI 门禁、发布与回滚说明。
- 不纳入本轮：全量架构重构、非阻塞类性能微调、外部依赖配置细节、产品级需求新增。
- 处理策略：优先解决“会导致功能误判、迁移失配、发布失稳”的事项，再推进结构性收敛。

### 3.1 目标状态

| 领域 | 当前状态 | 目标状态 |

| 主线 D：交付验证标准 | 固化 CI、测试、回滚、监控、PDF 交付模板 | 发布审批可直接引用 |

## 四、功能完整性补强方案

### 4.1 补强范围

| 范围 | 当前判断 | 治理动作 |
|------|----------|----------|
| Room Summary | 已有实现，但历史失败判断存在误判 | 对齐响应结构、补齐同步链验证、清除重复调用与脆弱断言 |
| Admin 核心路径 | 核心鉴权链存在，历史失败多由脚本误配触发 | 统一真实 admin token、路由路径、HTTP 方法与错误码断言 |
| DM / Space | 部分接口存在但前置数据不足、语义未完全稳定 | 引入 seed 流程，明确返回语义与空结果行为 |
| Thread / Verification | 目标路径与项目现有实现存在偏差 | 先完成接口映射，再决定标准兼容或私有保留 |
| 可选能力 | OIDC、TURN、Thirdparty、扩展 Federation 依赖外部配置 | 从主回归剥离为 optional suite |

### 4.2 实施要求

| 要求 | 说明 |
|------|------|
| 标准与私有分层 | Matrix 标准接口、Synapse Admin 接口、项目私有接口分别出清单，不混报通过率 |
| 断言升级 | 所有核心用例必须校验 HTTP status、关键字段、Matrix `errcode` 或项目错误码 |
| 前置种子化 | 设备、媒体、DM、Space、Pushers、Federation 目的端等均通过 seed 步骤准备 |
| 跳过原因标准化 | 仅允许使用“真实未实现/未收敛、前置不足、可选能力/外部依赖”三类 |
| 结果产物化 | 主回归必须输出失败与跳过清单，供 CI 与 PDF 审计引用 |

### 4.3 接口契约模板

| 字段 | 填写要求 |
|------|----------|
| 业务域 | 认证、房间、管理、联邦、媒体、Space、Thread 等明确归属 |
| 接口标识 | 使用“模块 + 动作”命名，不以脚本名代替业务名 |
| 协议级别 | Matrix 标准、Synapse Admin、项目私有扩展三选一 |
| 方法与路径 | 明确 HTTP 方法、版本前缀、路径参数与查询参数 |
| 鉴权模型 | 未认证、用户认证、管理员认证、服务间认证 |
| 请求约束 | 必填字段、字段类型、幂等性、顺序要求、前置资源 |
| 响应契约 | 成功状态码、必含字段、分页字段、空结果语义 |
| 失败契约 | 失败状态码、错误码、触发条件、可重试性 |
| 存储影响 | 涉及表、事务边界、事件写入、副作用 |
| 观测要求 | 日志字段、trace/span、metrics label、审计字段 |
| 验证要求 | 单元测试、集成测试、E2E、脚本回归、CI 门禁 |
| 文档归档 | 对应章节、版本号、变更记录、发布说明 |

### 4.4 状态机模板

| 字段 | 填写要求 |
|------|----------|
| 对象 | 房间摘要、设备验证、线程、任务、推送订阅等 |
| 状态集合 | 初始态、处理中、成功态、失败态、终态 |
| 触发事件 | 用户请求、后台任务、联邦回调、定时清理 |
| 守卫条件 | 权限、资源存在性、幂等校验、版本校验 |
| 持久化动作 | 写入表、更新索引、事件流或缓存 |
| 外部副作用 | 通知、日志、metrics、异步任务 |
| 超时/重试 | 允许的重试次数、退避策略、超时阈值 |
| 回滚路径 | 失败补偿、事务回滚、状态修正 |
| 观测字段 | request_id、user_id、room_id、state_before、state_after |
| 验收方式 | 状态迁移测试、异常路径测试、回归报告 |

### 4.5 异常码模板

| 字段 | 填写要求 |
|------|----------|
| 场景编号 | 按业务域唯一编号，支持映射到缺陷与告警 |
| HTTP 状态码 | 4xx/5xx 明确，不允许只描述“失败” |
| 协议错误码 | Matrix `M_*` 或项目扩展错误码 |
| 触发条件 | 鉴权失败、参数非法、资源不存在、状态冲突、外部依赖失败 |
| 返回信息 | 面向客户端的人类可读消息 |
| 可重试性 | 不可重试、客户端修正后重试、服务端退避重试 |
| 审计字段 | user_id、device_id、room_id、request_id、admin_id |
| 监控标签 | module、endpoint、errcode、status_class |
| 文档归属 | 所属接口契约与测试用例编号 |

## 五、Schema Drift 根治方案

### 5.1 根治目标

| 目标 | 当前基础 | 本轮要求 |
|------|----------|----------|
| 表覆盖闭环 | 已有 `check_schema_table_coverage.py` | 新增代码引用不得落入 exceptions 长期豁免 |
| 契约覆盖闭环 | 已有 `check_schema_contract_coverage.py` | 表存在之外，还要校验列、索引、约束与命名规范 |
| 完整性闭环 | 已接入 `pg_amcheck` 与逻辑 checksum | 发布准入必须引用产物，不再依赖口头确认 |
| 漂移检测闭环 | 已有 `drift-detection.yml` | 对齐当前迁移命名与目录策略，避免旧规则空转 |

### 5.2 治理规则

| 规则 | 说明 |
|------|------|
| 同批提交 | 任何结构变更必须同时提交 unified schema、增量迁移、回滚脚本、测试、门禁更新 |
| 先建契约再写 SQL | 先确认字段命名、索引、约束、外键，再编写迁移 |
| exceptions 只允许临时存在 | 每个例外项必须有编号、负责人、清理截止版本 |
| Fresh DB 与 Upgrade DB 双验证 | 统一 schema 建库与 `sqlx migrate run` 两条路径必须同时通过 |
| Drift 报告可阻断 | 出现 blocker 级漂移时直接阻断合并或发布 |

### 5.3 门禁矩阵

| 门禁 | 当前基础 | 目标动作 |
|------|----------|----------|
| Schema Table Coverage | 已上线 | 继续压缩例外清单，按业务批次清零 |
| Schema Contract Coverage | 已上线 | 扩展到关键索引、FK、唯一约束与命名规则 |
| Migration Layout Audit | 已上线 | 改为检查当前日期命名、回滚配套、归档状态 |
| Unified Schema Apply | 已上线 | 纳入所有关键补丁迁移并输出建库耗时、错误摘要 |
| sqlx Migrate Run | 已上线 | 按模块补充 smoke test 与 checksum 报告 |
| Drift Detection | 已上线 | 清理旧 Flyway 风格检查，统一到现行规则 |

## 六、迁移脚本优化合并与冗余脚本清理

### 6.1 现状判断

| 项目 | 当前实情 | 风险 |
|------|----------|------|
| 统一基线 | `00000000_unified_schema_v6.sql` 已作为基线 | 新增结构若只补增量不补基线，会再次产生 fresh DB 漂移 |
| 综合增量 | `99999999_unified_incremental_migration.sql` 已存在 | 若长期与根目录分散迁移并存，会出现入口歧义 |
| 历史迁移 | 根目录保留大量日期迁移 | 相同主题分散、审计成本高 |
| 回滚脚本 | `rollback/` 已建立 | 并非所有历史迁移都已完成配套治理 |
| 目录治理 | `incremental/`、`hotfix/`、`archive/` 已建立 | 规则已定义，执行口径仍需固化到发布流程 |

### 6.2 合并策略

| 阶段 | 动作 | 产出 |
|------|------|------|
| 阶段 A | 以当前 unified schema 作为新环境唯一基线 | 新环境只依赖基线 + 当期正式增量 |
| 阶段 B | 将同主题碎片迁移按业务域收敛为批次增量 | thread、retention、room summary、space、notification 等形成清晰批次 |
| 阶段 C | 对已被 unified schema 吸收且不再需要单独入口的历史脚本建立归档清单 | 形成“保留、归档、废弃、待核销”四类台账 |
| 阶段 D | 统一综合增量脚本定位 | 明确其为交付便利产物还是正式升级入口，避免双重权威 |

### 6.3 冗余脚本清理标准

| 判定条件 | 清理动作 |
|----------|----------|
| 已被 unified schema 完整吸收，且无独立升级价值 | 迁入 `archive/` 并在索引文档标记替代关系 |
| 与后续批次迁移重复定义同一表/索引 | 合并后保留最新正式脚本，旧脚本归档 |
| 仅用于历史热修复，且已纳入正式迁移 | 从 `hotfix/` 收敛并补发布说明 |
| 缺回滚、缺测试、缺说明 | 不允许进入正式迁移链 |

### 6.4 迁移资产台账模板

| 字段 | 填写要求 |
|------|----------|
| 脚本名称 | 文件名与业务域 |
| 脚本类型 | 基线、增量、回滚、热修复、归档 |
| 生效范围 | 新环境、升级环境、紧急修复 |
| 替代关系 | 被哪个脚本或基线吸收 |
| 回滚方式 | 独立回滚、批次回滚、不可逆声明 |
| 关联测试 | smoke、roundtrip、schema contract、checksum |
| 归档状态 | 活跃、待归档、已归档、待删除 |

## 七、测试、CI 与交付验证标准

### 7.1 测试分层标准

| 层级 | 当前基础 | 本轮标准 |
|------|----------|----------|
| Unit | `tests/unit` 已覆盖大量存储与服务逻辑 | 关键模块必须补齐异常路径与状态机迁移测试 |
| Integration | `tests/integration` 已覆盖 API 主路径 | 改为以接口契约为断言标准，强校验状态码与错误码 |
| E2E | `tests/e2e` 已存在用户流程 | 仅保留关键业务价值链，用于发布前放行 |
| Performance | `tests/performance` 已存在基准框架 | 纳入 P95/P99、失败率、资源快照产物 |
| Script Regression | `scripts/api-integration_test.sh` 已能输出失败/跳过产物 | 作为 API 契约回归层，不再承担全部实现真相判断 |

### 7.2 CI 质量门禁

| 类别 | 当前工作流 | 交付标准 |
|------|------------|----------|
| 代码质量 | `ci.yml` | `cargo fmt --all -- --check`、`cargo clippy --all-features --locked -- -D warnings`、`cargo test --all-features --locked` 全通过 |
| 数据库门禁 | `db-migration-gate.yml` | unified schema、增量迁移、contract coverage、pg_amcheck、checksum 全通过 |
| 文档质量 | `docs-quality-gate.yml` | 扩展到顶层实施文档与数据库治理文档，不再只校验单一报告 |
| 漂移检测 | `drift-detection.yml` | 规则与现行迁移规范一致，发现 blocker 直接阻断 |
| 产物归档 | 测试、校验、报告类工作流 | 失败清单、跳过清单、checksum、amcheck、drift report 统一归档 |

### 7.3 交付验收标准

| 维度 | 验收标准 |
|------|----------|
| 功能完整性 | 核心 Matrix Client API、Synapse Admin API 关键链路通过，误配项清零后再统计真实缺口 |
| Schema Drift | 代码引用表集合、字段契约、索引契约无 blocker；exceptions 有明确退场计划 |
| 迁移治理 | 新环境建库与升级环境迁移均通过；回滚脚本完成演练 |
| 回归质量 | 主回归脚本输出失败/跳过明细；跳过原因符合分类规范 |

## 九、风险、回滚与监控方案

### 9.1 风险矩阵

| 风险 | 触发信号 | 影响 | 应对策略 |
|------|----------|------|----------|
| 路由契约调整引发回归 | 核心 API 状态码或字段变化 | 客户端兼容性下降 | 契约评审先行，发布前跑完整回归 |
| 迁移合并引发升级失败 | `sqlx migrate run` 失败或耗时激增 | 升级阻塞 | 保留批次化回滚、先灰度后全量 |
| Drift 门禁过严导致开发阻塞 | 新表/索引改动频繁被拦截 | 研发效率下降 | 允许短期 exceptions，但必须附负责人和截止版本 |
| 测试脚本误报残留 | 失败项与实现不一致 | 优先级误判 | 将脚本产物与契约表双向核对 |
| 文档与实现再次脱节 | 交付材料无法支撑发布审批 | 审批反复 | 将文档更新纳入 Definition of Done |

### 9.2 回滚策略

| 场景 | 回滚动作 | 证据要求 |
|------|----------|----------|
| 单批次增量迁移失败 | 执行对应 rollback 脚本并恢复发布前备份 | 回滚日志、影响表清单、责任人确认 |
| unified schema 新环境异常 | 停止发布，回退到上一个稳定基线镜像 | 建库失败日志、修复 PR、重新验收结果 |
| API 契约回归失败 | 回滚相关 handler/service 变更，保留数据库兼容层 | 回归报告、错误码统计、兼容说明 |
| 监控异常放大 | 触发限流、降级或停止灰度 | 监控截图、告警事件、处置记录 |

### 9.3 监控指标

| 指标 | 阈值建议 | 来源 |
|------|----------|------|
| API 5xx 比例 | 核心链路持续高于基线即告警 | 应用日志与 metrics |
| Matrix `errcode` 分布 | 新增高频 `M_UNKNOWN`、`M_FORBIDDEN` 需复核 | 回归产物与应用日志 |
| 迁移执行耗时 | 超过历史基线 20% 进入人工复核 | CI 报告与发布演练 |
| Schema 异常数 | drift blocker > 0 直接阻断 | drift report |
| 数据完整性异常 | `pg_amcheck` 或 logical checksum 失败 | DB 门禁产物 |

## 十一、结论与执行要求

本项目后端优化的关键不在于继续堆叠零散修补项，而在于把“接口契约、状态机、Schema、迁移、测试、CI、回滚、PDF 交付”收敛成一套统一执行面。后续所有优化任务均应按本方案拆分为可验证工作包，做到需求有契约、实现有门禁、发布有证据、异常可回滚。

### 11.1 发布前必须满足

| 条件 | 要求 |
|------|------|
| 功能 | P0 项全部完成，P1 无阻塞项 |
| 测试 | 核心回归、集成测试、数据库 smoke tests 全通过 |
| 数据库 | unified schema、增量迁移、contract gate、checksum、amcheck 全通过 |
| 文档 | 本方案、变更记录、回滚说明、PDF 版本同步完成 |
| 审批 | 后端、DBA、QA 至少双人复核通过 |

### 11.2 后续维护要求

- 每次新增接口必须先补接口契约模板，再进入开发。
- 每次新增状态流必须补状态机模板与异常码模板。
- 每次新增表或索引必须同步更新 unified schema、增量迁移、回滚脚本与门禁。
- 每次发版必须输出 Markdown 与 PDF 两份正式交付物。
