## 一、文档定位

本方案用于替换现有偏“问题罗列式”的优化计划，形成可用于评审、排期、交付与 PDF 归档的正式《后端优化实施方案》章节草案。方案遵循 Matrix/Synapse 风格：以接口契约为边界、以服务分层为实施单元、以迁移闭环和回归证据为发布准入。

### 1.1 目标定义

| 目标              | 说明                     | 结果要求              |
| :-------------- | :--------------------- | :---------------- |
| 功能完整性补强         | 修正测试脚本与真实实现偏差，补齐未收敛能力  | 核心链路按契约稳定通过       |
| Schema Drift 根治 | 消除“代码已依赖、Schema 未闭环”问题 | 代码、迁移、Schema、门禁统一 |
| 迁移脚本优化合并        | 形成统一基线与增量治理节奏          | 新环境、升级环境均可重复执行    |
| 冗余脚本清理          | 下线重复、过时、已被吸收的历史脚本      | 迁移目录可审计、可维护       |
| 交付验证标准化         | 将验收标准前移到测试与 CI         | 发布前具备完整证据链        |

### 1.2 设计原则

| 原则            | 要求                                                            |
| :------------ | :------------------------------------------------------------ |
| 契约优先          | 先定义路径、权限、状态码、错误码、响应字段，再落实现有 handler 与 service 收敛              |
| 分层收口          | `web/routes` 负责协议与鉴权，`services` 负责业务编排，`storage` 负责 SQL 与事务边界 |
| Schema 单一真实来源 | 统一 schema、增量迁移、回滚脚本、门禁脚本必须同批更新                                |
| Synapse 风格兼容  | 核心 Matrix Client API 与 Synapse Admin API 分层治理，不混用私有扩展与标准路径    |
| 证据可追溯         | 测试、迁移、校验、回滚、监控都必须有产物或记录可审计                                    |

### 2.2 主要问题归类

| 类别          | 现象                                          | 根因                                | 严重度 |
| :---------- | :------------------------------------------ | :-------------------------------- | :-- |
| 功能契约不一致     | API 测试脚本与真实路由、方法、鉴权、返回结构存在偏差                | 规范、实现、回归脚本未同步维护                   | P0  |
| Schema 闭环缺失 | 历史上存在代码已读写、迁移未建表或索引不齐                       | 统一 schema、增量迁移、exceptions 清理节奏不同步 | P0  |
| 迁移资产膨胀      | 根目录历史脚本、统一增量脚本、目录治理并存                       | 迁移整合完成一半，归档口径未完全统一                | P1  |
| 回归标准偏宽      | 大量用例只做关键字探测，未强校验 HTTP 状态码与 Matrix `errcode` | 测试目标更像探测脚本，而非正式验收套件               | P1  |
| 文档与 CI 脱节   | 实施文档、门禁范围、发布准入没有形成一体化说明                     | 文档尚未升级为工程交付物                      | P2  |

### 2.3 本轮优化边界

- 纳入本方案：核心 Matrix Client API、Synapse Admin API、Schema/迁移治理、测试分层、CI 门禁、发布与回滚说明。
- 不纳入本轮：全量架构重构、非阻塞类性能微调、外部依赖配置细节、产品级需求新增。
- 处理策略：优先解决“会导致功能误判、迁移失配、发布失稳”的事项，再推进结构性收敛。

### 2.4 验证指出的缺口汇总

## 四、功能完整性补强方案

### 4.1 补强范围

| 范围                    | 当前判断                                      | 治理动作                                |
| :-------------------- | :---------------------------------------- | :---------------------------------- |
| Room Summary          | 已有实现，但历史失败判断存在误判                          | 对齐响应结构、补齐同步链验证、清除重复调用与脆弱断言          |
| Admin 核心路径            | 核心鉴权链存在，历史失败多由脚本误配触发                      | 统一真实 admin token、路由路径、HTTP 方法与错误码断言 |
| DM / Space            | 部分接口存在但前置数据不足、语义未完全稳定                     | 引入 seed 流程，明确返回语义与空结果行为             |
| Thread / Verification | 目标路径与项目现有实现存在偏差                           | 先完成接口映射，再决定标准兼容或私有保留                |
| 可选能力                  | OIDC、TURN、Thirdparty、扩展 Federation 依赖外部配置 | 从主回归剥离为 optional suite              |

### 4.2 实施要求

| 要求      | 说明                                                  |
| :------ | :-------------------------------------------------- |
| 标准与私有分层 | Matrix 标准接口、Synapse Admin 接口、项目私有接口分别出清单，不混报通过率     |
| 断言升级    | 所有核心用例必须校验 HTTP status、关键字段、Matrix `errcode` 或项目错误码 |
| 前置种子化   | 设备、媒体、DM、Space、Pushers、Federation 目的端等均通过 seed 步骤准备 |
| 跳过原因标准化 | 仅允许使用“真实未实现/未收敛、前置不足、可选能力/外部依赖”三类                   |
| 结果产物化   | 主回归必须输出失败与跳过清单，供 CI 与 PDF 审计引用                      |

### 4.3 缺失核心功能完整清单

| 功能   | 当前缺口                             | 最小交付                  | 放行条件                     |
| :--- | :------------------------------- | :-------------------- | :----------------------- |
| 权限审计 | 缺少统一的敏感操作审计流，权限变更与管理动作难追溯        | 审计写入、查询、追溯、导出闭环       | 审计事件可查、不可静默丢失、支持按用户与资源过滤 |
| 批量操作 | 管理与运维类接口以单资源操作为主，批量任务缺少异步编排与结果汇总 | 批量提交、任务查询、失败重试、部分成功回执 | 批量任务具备状态机、幂等键、失败明细与取消能力  |
| 数据导出 | 缺少面向运维/审计的数据导出契约，导出任务与权限边界不清晰    | 导出申请、异步生成、下载、过期清理     | 导出带签名 URL、审计留痕、过期回收与配额控制 |
| 监控告警 | 指标存在但告警收敛与升级路径不完整                | 指标采集、阈值告警、抑制、恢复通知     | 关键指标有 owner、有升级链路、有演练记录  |
| 灰度开关 | 缺少面向功能与迁移的统一灰度开关治理               | 开关注册、范围投放、审计、过期开关清理   | 开关变更可回溯、支持按租户/房间/用户分层灰度  |

### 4.4 核心功能模板条目

#### 4.4.1 权限审计

| 类别   | 模板条目                                                                                                                                                                                                                                 |
| :--- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 接口契约 | `POST /_synapse/admin/v1/audit/events` 写入审计事件，`GET /_synapse/admin/v1/audit/events` 查询，`GET /_synapse/admin/v1/audit/events/:event_id` 追溯详情；鉴权要求为管理员认证，必填字段含 `actor_id`、`action`、`resource_type`、`resource_id`、`result`、`request_id` |
| 状态机  | `received -> persisted -> indexed -> queryable -> archived`，当索引失败时转入 `persisted_pending_index`，后台补偿后回到 `queryable`，归档后进入 `archived` 终态                                                                                               |
| 异常码  | `AUDIT_FORBIDDEN`、`AUDIT_INVALID_ACTION`、`AUDIT_STORAGE_FAILED`、`AUDIT_EXPORT_CONFLICT`，分别映射权限不足、非法动作、存储失败、并发导出冲突                                                                                                                    |
| 性能指标 | 单条写入 P99 < 100ms；查询分页 P99 < 300ms；单日审计 1000 万行时按 `actor_id + created_ts` 检索 30 秒内返回首屏；超阈值触发索引复核与冷热分层                                                                                                                                 |

#### 4.4.2 批量操作

| 类别   | 模板条目                                                                                                                                                                                                                    |
| :--- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 接口契约 | `POST /_synapse/admin/v1/batch/jobs` 创建任务，`GET /_synapse/admin/v1/batch/jobs/:job_id` 查询进度，`POST /_synapse/admin/v1/batch/jobs/:job_id/cancel` 取消任务；请求必须带 `idempotency_key`、`operation_type`、`targets[]`、`requested_by` |
| 状态机  | `draft -> queued -> running -> partial_success/succeeded/failed -> archived`，取消路径为 `queued/running -> cancel_requested -> cancelled`                                                                                    |
| 异常码  | `BATCH_INVALID_TARGET`、`BATCH_TOO_LARGE`、`BATCH_DUPLICATE_REQUEST`、`BATCH_CONFLICT`、`BATCH_TIMEOUT`                                                                                                                     |
| 性能指标 | 单任务目标数默认上限 1000；排队入列 P99 < 200ms；1000 个目标的批量任务在 5 分钟内完成并返回失败明细；失败率 > 1% 自动进入熔断评估                                                                                                                                        |

#### 4.4.3 数据导出

| 类别   | 模板条目                                                                                                                                                                                                    |
| :--- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 接口契约 | `POST /_synapse/admin/v1/exports` 提交导出，`GET /_synapse/admin/v1/exports/:export_id` 查询，`GET /_synapse/admin/v1/exports/:export_id/download` 下载；请求需声明 `scope`、`format`、`filters`、`expires_at`，下载必须验证一次性签名 |
| 状态机  | `requested -> authorized -> generating -> ready -> downloaded/expired -> purged`，导出失败进入 `failed`，可人工重试回到 `authorized`                                                                                   |
| 异常码  | `EXPORT_FORBIDDEN`、`EXPORT_SCOPE_INVALID`、`EXPORT_RATE_LIMITED`、`EXPORT_FILE_EXPIRED`、`EXPORT_GENERATION_FAILED`                                                                                        |
| 性能指标 | 100 万行导出生成 P99 < 10 分钟；导出文件下载首字节 < 2 秒；并发导出任务数达到阈值时进入排队并发控制，不允许拖垮主交易链路                                                                                                                                  |

#### 4.4.4 监控告警

| 类别   | 模板条目                                                                                                                                                                         |
| :--- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 接口契约 | `GET /_synapse/admin/v1/telemetry/health` 提供健康态，`GET /_synapse/admin/v1/telemetry/alerts` 查询告警，`POST /_synapse/admin/v1/telemetry/alerts/:alert_id/ack` 确认告警；需要管理员认证或内部服务间认证 |
| 状态机  | `ok -> warning -> critical -> acknowledged -> recovered -> closed`，恢复前禁止直接关闭 critical 告警                                                                                     |
| 异常码  | `ALERT_RULE_NOT_FOUND`、`ALERT_ACK_FORBIDDEN`、`ALERT_SUPPRESSION_ACTIVE`、`ALERT_DELIVERY_FAILED`                                                                              |
| 性能指标 | 指标采集周期 15 秒；关键告警从触发到送达 < 60 秒；连续 3 个采样窗口超阈值触发升级；恢复通知 < 120 秒                                                                                                                 |

#### 4.4.5 灰度开关

| 类别   | 模板条目                                                                                                                                                                                                                                  |
| :--- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 接口契约 | `POST /_synapse/admin/v1/feature-flags` 创建开关，`PATCH /_synapse/admin/v1/feature-flags/:flag_key` 调整比例或目标，`GET /_synapse/admin/v1/feature-flags/:flag_key` 查询详情；请求需包含 `flag_key`、`target_scope`、`rollout_percent`、`expires_at`、`reason` |
| 状态机  | `draft -> scheduled -> active -> ramping -> fully_enabled/rolled_back -> removed`，临时开关到期未清理时进入 `expired_pending_removal`                                                                                                              |
| 异常码  | `FLAG_INVALID_SCOPE`、`FLAG_EXPIRED`、`FLAG_CONFLICT`、`FLAG_GUARDRAIL_BLOCKED`、`FLAG_ROLLBACK_REQUIRED`                                                                                                                                 |
| 性能指标 | 开关读取 P99 < 20ms；配置传播 < 30 秒；灰度切换后 5 分钟内完成护栏指标比对；触发 guardrail 时 1 分钟内自动回滚                                                                                                                                                              |

### 4.5 接口契约模板

| 字段    | 填写要求                               |
| :---- | :--------------------------------- |
| 业务域   | 认证、房间、管理、联邦、媒体、Space、Thread 等明确归属  |
| 接口标识  | 使用“模块 + 动作”命名，不以脚本名代替业务名           |
| 协议级别  | Matrix 标准、Synapse Admin、项目私有扩展三选一  |
| 方法与路径 | 明确 HTTP 方法、版本前缀、路径参数与查询参数          |
| 鉴权模型  | 未认证、用户认证、管理员认证、服务间认证               |
| 请求约束  | 必填字段、字段类型、幂等性、顺序要求、前置资源            |

## 五、Schema Drift 根治方案

### 5.1 根治目标

| 目标     | 当前基础                                   | 本轮要求                       |
| :----- | :------------------------------------- | :------------------------- |
| 表覆盖闭环  | 已有 `check_schema_table_coverage.py`    | 新增代码引用不得落入 exceptions 长期豁免 |
| 契约覆盖闭环 | 已有 `check_schema_contract_coverage.py` | 表存在之外，还要校验列、索引、约束与命名规范     |
| 完整性闭环  | 已接入 `pg_amcheck` 与逻辑 checksum          | 发布准入必须引用产物，不再依赖口头确认        |
| 漂移检测闭环 | 已有 `drift-detection.yml`               | 对齐当前迁移命名与目录策略，避免旧规则空转      |

### 5.2 治理规则

| 规则                        | 说明                                            |
| :------------------------ | :-------------------------------------------- |
| 同批提交                      | 任何结构变更必须同时提交 unified schema、增量迁移、回滚脚本、测试、门禁更新 |
| 先建契约再写 SQL                | 先确认字段命名、索引、约束、外键，再编写迁移                        |
| exceptions 只允许临时存在        | 每个例外项必须有编号、负责人、清理截止版本                         |
| Fresh DB 与 Upgrade DB 双验证 | 统一 schema 建库与 `sqlx migrate run` 两条路径必须同时通过   |
| Drift 报告可阻断               | 出现 blocker 级漂移时直接阻断合并或发布                      |

### 5.3 门禁矩阵

| 门禁                       | 当前基础 | 目标动作                                           |
| :----------------------- | :--- | :--------------------------------------------- |
| Schema Table Coverage    | 已上线  | 继续压缩例外清单，按业务批次清零                               |
| Schema Contract Coverage | 已上线  | 扩展到关键索引、FK、唯一约束与命名规则                           |
| Migration Layout Audit   | 已上线  | 改为检查 `V{版本}__{Jira编号}_{简短描述}.sql` 命名、回滚配套、归档状态 |
| Unified Schema Apply     | 已上线  | 纳入所有关键补丁迁移并输出建库耗时、错误摘要                         |
| sqlx Migrate Run         | 已上线  | 按模块补充 smoke test 与 checksum 报告                 |
| Drift Detection          | 已上线  | 清理旧 Flyway 风格检查，统一到现行规则                        |

## 六、迁移脚本优化合并与冗余脚本清理

### 6.1 现状判断

| 项目   | 当前实情                                             | 风险                              |
| :--- | :----------------------------------------------- | :------------------------------ |
| 统一基线 | `00000000_unified_schema_v6.sql` 已作为基线           | 新增结构若只补增量不补基线，会再次产生 fresh DB 漂移 |
| 综合增量 | `99999999_unified_incremental_migration.sql` 已存在 | 若长期与根目录分散迁移并存，会出现入口歧义           |
| 历史迁移 | 根目录保留大量日期迁移                                      | 相同主题分散、审计成本高                    |
| 回滚脚本 | `rollback/` 已建立                                  | 并非所有历史迁移都已完成配套治理                |
| 目录治理 | `incremental/`、`hotfix/`、`archive/` 已建立          | 规则已定义，执行口径仍需固化到发布流程             |

### 6.2 baseline / migrate / undo 三段式定义

| 阶段       | 定义                                   | 输入                                                    | 输出                  | 验证要求                                             |
| :------- | :----------------------------------- | :---------------------------------------------------- | :------------------ | :----------------------------------------------- |
| baseline | 新环境初始化基线，只包含一个时点可完整建库的权威 schema      | `00000000_unified_schema_v6.sql` 与版本化 schema manifest | 空库可用的完整结构、索引、约束、默认值 | 需在全新 PostgreSQL 实例中一次执行成功，并记录建库耗时、DDL 摘要、SHA-256 |
| migrate  | 升级环境增量变更链，只表达从已上线版本到目标版本的结构或数据演进     | `migrations/incremental/*.sql`、必要 `hotfix` 收敛脚本       | 可按版本连续执行的增量链路       | 需在升级样本库执行通过，校验 schema diff、数据校验、耗时、锁等待           |
| undo     | 与增量变更一一对应的回退能力，负责在灰度或正式发布失败时恢复到前一稳定态 | `migrations/rollback/*.rollback.sql`、备份清单、回滚 runbook  | 可审计的回滚动作、影响范围与恢复证据  | 需在 staging 演练完成，5 分钟目标内拿到可用结果并产出回滚记录             |

### 6.3 合并策略

| 阶段   | 动作                                        | 产出                                                       |
| :--- | :---------------------------------------- | :------------------------------------------------------- |
| 阶段 A | 以当前 unified schema 作为新环境唯一基线              | 新环境只依赖基线 + 当期正式增量                                        |
| 阶段 B | 将同主题碎片迁移按业务域收敛为批次增量                       | thread、retention、room summary、space、notification 等形成清晰批次 |
| 阶段 C | 对已被 unified schema 吸收且不再需要单独入口的历史脚本建立归档清单 | 形成“保留、归档、废弃、待核销”四类台账                                     |
| 阶段 D | 统一综合增量脚本定位                                | 明确其为交付便利产物还是正式升级入口，避免双重权威                                |

### 6.4 冗余脚本清理标准

| 判定条件                            | 清理动作                       |
| :------------------------------ | :------------------------- |
| 已被 unified schema 完整吸收，且无独立升级价值 | 迁入 `archive/` 并在索引文档标记替代关系 |
| 与后续批次迁移重复定义同一表/索引               | 合并后保留最新正式脚本，旧脚本归档          |
| 仅用于历史热修复，且已纳入正式迁移               | 从 `hotfix/` 收敛并补发布说明       |
| 缺回滚、缺测试、缺说明                     | 不允许进入正式迁移链                 |

### 6.5 命名、防篡改与发布校验

| 项目        | 标准                                                                                 |
| :-------- | :--------------------------------------------------------------------------------- |
| 正式迁移命名    | 所有正式迁移统一命名为 `V{版本}__{Jira编号}_{简短描述}.sql`，例如 `V6_0_5__MIG-142_add_audit_events.sql` |
| down 脚本命名 | 对应回滚脚本命名为 `V{版本}__{Jira编号}_{简短描述}.down.sql` 或 `.rollback.sql`，二者必须择一并在索引中统一        |
| manifest  | 每次发布生成 `migrations/MANIFEST-{release}.txt`，记录文件名、字节大小、SHA-256、owner、适用阶段           |
| 防篡改校验     | CI 与发布脚本必须对 manifest 中所有 SQL 文件重算 SHA-256；任一不一致直接阻断                                |
| 追溯绑定      | manifest 必须记录 Git commit、发布单号、Jira 编号与制品仓 URL，确保文件、代码、审批单三向追溯                      |

### 6.6 大表性能基线与超阈值处理

| 项目      | 基线要求                                    | 超阈值处理                           |
| :------ | :-------------------------------------- | :------------------------------ |
| 大表迁移数据量 | 以 1000 万行为标准样本，覆盖主键、索引、回填、校验场景          | 样本不足时不得宣称生产级通过，需补齐 staging 数据回放 |
| 执行窗口    | 单批次大表迁移 30 秒内完成主要 DDL/DML 窗口，超过即判定为风险变更 | 自动降级为分批迁移、在线回填、延迟索引创建或暂停发布      |
| 锁等待     | 单次阻塞锁等待 < 3 秒                           | 超过阈值立即终止迁移并切换只读观测，复核 SQL 与索引策略  |
| 资源占用    | CPU < 70%，IO 等待 < 20%，连接池占用 < 80%       | 任一项超限触发 SRE 告警并停止继续执行后续迁移       |
| 验证产物    | 输出执行耗时、影响行数、EXPLAIN/ANALYZE 摘要、资源快照     | 产物缺失按未通过处理，禁止用“人工观察正常”替代        |

### 6.7 down 脚本标准与 staging 回滚演练

| 项目        | 标准                                            |
| :-------- | :-------------------------------------------- |
| down 脚本结构 | 必须包含适用 up 脚本、前置检查、幂等判断、回滚 SQL、回滚后验证 SQL、不可逆声明 |
| 幂等要求      | 重复执行不得报致命错误；对不存在对象必须使用条件判断或 `IF EXISTS`       |
| 数据安全      | 涉及删列、删表、覆写数据的 down 脚本必须引用备份点或显式声明不可逆          |
| 演练环境      | staging 必须使用接近生产的数据量级、索引规模与副本拓扑，不允许仅在空库演练     |
| 时间目标      | staging 回滚演练需在 5 分钟内完成“发现异常、执行回滚、恢复健康检查通过”全流程 |

staging 回滚演练步骤：

1. 记录当前目标版本、最近一次稳定版本、Git commit、迁移 manifest SHA-256。
2. 对 staging 执行发布前备份，生成结构备份、关键表数据备份与 checksum 快照。
3. 执行目标版本 baseline/migrate 验证，确认异常触发条件可稳定复现。
4. 按版本逆序执行对应 down 或 rollback 脚本，同时记录开始时间、结束时间、影响对象与异常输出。
5. 运行健康检查、核心 API smoke、schema contract、logical checksum，确认回到发布前稳定态。
6. 归档回滚耗时、责任人、失败点、修复动作，并把演练结果挂入发布审批附件。

### 6.8 生命周期标签、保留窗口与删除审计

| 标签           | 含义                 | 保留窗口           | 删除要求                          |
| :----------- | :----------------- | :------------- | :---------------------------- |
| `deprecated` | 已有替代脚本，仍处于兼容保留期    | 至少保留 2 个发布周期   | 删除前需完成替代关系验证与 DBA 审核          |
| `unused`     | 当前链路不再执行，但仍需保留审计证据 | 至少保留 1 个发布周期   | 删除前需确认不在任何部署流程、CI、runbook 被引用 |
| `test-only`  | 仅用于测试环境、演练或基准生成    | 可按测试策略保留 1 个周期 | 删除前需确认测试工单或基准场景已迁移            |

删除审计要求：

- 每次物理删除必须先写入 `docs/CHANGELOG-DB.md`，记录删除原因、替代脚本、审批人、归档位置。
- 删除提交必须附 Git 归档 tag、SHA-256 manifest、staging 验证结果与回滚可恢复说明。
- 删除后 24 小时内执行一次仓库引用扫描，确认 README、runbook、CI、脚本无死链。

### 6.9 Git 归档备份、追溯 tag 与紧急恢复

| 项目       | 标准                                                                   |
| :------- | :------------------------------------------------------------------- |
| Git 归档备份 | 每批次迁移清理前创建 `archive/{release}/{jira}` tag，并导出对应 SQL 与 manifest 压缩包   |
| 追溯 tag   | tag 必须绑定发布版本、Jira、责任人、日期，格式如 `archive/v6.0.5/MIG-142`                |
| 紧急恢复     | 恢复优先级为 Git tag -> 制品归档 -> 结构备份 -> 数据备份；任一恢复动作都需记录 commit、SHA-256、执行人 |
| 恢复演练     | 每个季度至少演练一次“从归档 tag 恢复迁移 + 重放回滚脚本 + 校验 checksum”                      |
| 审计证据     | 恢复后必须比对文件 SHA-256、`schema_migrations` 状态与健康检查，形成恢复报告                 |

### 6.10 迁移资产台账模板

## 八、P0 / P1 / P2 排期与人力时序估算

### 8.1 优先级定义

| 优先级 | 定义                              | 放行策略              |
| :-- | :------------------------------ | :---------------- |
| P0  | 阻塞发布、造成核心功能误判、可能导致 Schema/数据硬失败 | 未完成不得发布           |
| P1  | 显著影响稳定性、兼容性、迁移治理效率              | 可灰度但不得长期积压        |
| P2  | 可维护性、文档、流程优化项                   | 纳入后续迭代但需有明确 owner |

### 8.2 排期总览

## 九、风险、回滚与监控方案

### 9.1 风险矩阵

| 风险               | 触发信号                       | 影响       | 应对策略                         |
| :--------------- | :------------------------- | :------- | :--------------------------- |
| 路由契约调整引发回归       | 核心 API 状态码或字段变化            | 客户端兼容性下降 | 契约评审先行，发布前跑完整回归              |
| 迁移合并引发升级失败       | `sqlx migrate run` 失败或耗时激增 | 升级阻塞     | 保留批次化回滚、先灰度后全量               |
| Drift 门禁过严导致开发阻塞 | 新表/索引改动频繁被拦截               | 研发效率下降   | 允许短期 exceptions，但必须附负责人和截止版本 |
| 测试脚本误报残留         | 失败项与实现不一致                  | 优先级误判    | 将脚本产物与契约表双向核对                |
| 文档与实现再次脱节        | 交付材料无法支撑发布审批               | 审批反复     | 将文档更新纳入 Definition of Done   |

### 9.2 回滚策略

| 场景                   | 回滚动作                             | 证据要求                |
| :------------------- | :------------------------------- | :------------------ |
| 单批次增量迁移失败            | 执行对应 rollback 脚本并恢复发布前备份         | 回滚日志、影响表清单、责任人确认    |
| unified schema 新环境异常 | 停止发布，回退到上一个稳定基线镜像                | 建库失败日志、修复 PR、重新验收结果 |
| API 契约回归失败           | 回滚相关 handler/service 变更，保留数据库兼容层 | 回归报告、错误码统计、兼容说明     |
| 监控异常放大               | 触发限流、降级或停止灰度                     | 监控截图、告警事件、处置记录      |

### 9.3 监控指标

| 指标                  | 阈值建议                               | 来源                     |
| :------------------ | :--------------------------------- | :--------------------- |
| API 5xx 比例          | 核心链路持续高于基线即告警                      | 应用日志与 metrics          |
| Matrix `errcode` 分布 | 新增高频 `M_UNKNOWN`、`M_FORBIDDEN` 需复核 | 回归产物与应用日志              |
| 迁移执行耗时              | 超过历史基线 20% 进入人工复核                  | CI 报告与发布演练             |
| Schema 异常数          | drift blocker > 0 直接阻断             | drift report           |
| 数据完整性异常             | `pg_amcheck` 或 logical checksum 失败 | DB 门禁产物                |
| 灰度护栏                | 5 分钟窗口内 QPS、P99、错误率任一越线即自动回滚       | 应用 metrics、SRE 告警、开关平台 |

## 十二、结论与执行要求
  
本项目后端优化的关键不在于继续堆叠零散修补项，而在于把“接口契约、状态机、Schema、迁移、测试、CI、回滚、PDF 交付”收敛成一套统一执行面。后续所有优化任务均应按本方案拆分为可验证工作包，做到需求有契约、实现有门禁、发布有证据、异常可回滚。

### 12.1 发布前必须满足

| 条件  | 要求                                                     |
| :-- | :----------------------------------------------------- |
| 功能  | P0 项全部完成，P1 无阻塞项                                       |
| 测试  | 单元测试覆盖率 ≥ 80%，新增接口集成测试全覆盖，核心回归、数据库 smoke tests 全通过     |
| 数据库 | unified schema、增量迁移、contract gate、checksum、amcheck 全通过 |
| 性能  | QPS 承载能力达到现峰值 120% 负载目标，P99 与错误率达标，1000 万行/30 秒迁移基线通过  |
| 文档  | 本方案、变更记录、回滚说明、PDF 版本同步完成，manifest SHA-256 与归档 tag 可追溯  |
| 审批  | QA、DBA、SRE 三方签字准出完成，后端 owner 对缺口闭环结果负责                 |

### 12.2 后续维护要求

- 每次新增接口必须先补接口契约模板，再进入开发。
- 每次新增状态流必须补状态机模板与异常码模板。
- 每次新增表或索引必须同步更新 unified schema、增量迁移、回滚脚本与门禁。
- 每次发版必须输出 Markdown 与 PDF 两份正式交付物。
- 每次迁移发布必须生成 manifest、SHA-256、归档 tag、staging 回滚演练记录。
- 每次归档或删除脚本必须打上 `deprecated`、`unused`、`test-only` 生命周期标签并完成删除审计。
