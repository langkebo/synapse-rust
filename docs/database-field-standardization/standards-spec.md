# 数据库字段标准化规范 v2

## 1. 命名规则

- 全部字段使用 snake_case
- 主键统一使用 `id`（仅在单表内部语义）
- 业务标识统一使用 `*_id` 后缀，如 `user_id`、`room_id`
- 布尔字段统一使用 `is_` 或 `has_` 前缀
- 时间字段统一使用 `*_ts`（毫秒时间戳，BIGINT）
- JSON 结构化字段统一使用 `JSONB`

## 2. 类型规则

- 关联 ID 默认使用 `VARCHAR(255)`，同语义字段跨表保持一致
- 时间戳统一使用 `BIGINT`
- 状态与枚举默认使用 `VARCHAR(64)` 或 `VARCHAR(128)`，避免裸 `TEXT`
- 大文本字段使用 `TEXT`
- 可扩展元数据使用 `JSONB`

## 3. 约束规则

- 每张业务表至少具备主键
- 关联字段优先建立外键；无法建立外键时必须在文档中登记原因
- 高频查询字段必须建立索引
- 非空业务关键字段显式 `NOT NULL`
- 语义一致字段使用 `CHECK` 约束保障一致性

## 4. 冗余字段治理

- 冗余字段进入三阶段治理：兼容期、双写校验期、清理期
- 兼容期保留旧字段并建立同步规则
- 双写校验期通过约束或作业校验一致性
- 清理期删除旧字段并更新全部 SQL 与代码引用

## 5. 模块映射原则

- 以领域为单位定义“规范字段词典”
- 同领域 ID 字段命名与类型必须一致
- 跨领域引用需提供字段映射表与迁移顺序
- 不允许新增未登记字段别名

## 6. 审核门禁

- 新增迁移必须附带字段标准检查结果
- 代码评审必须包含字段命名、类型、约束三项检查
- CI 必须执行字段审计脚本并输出报告
