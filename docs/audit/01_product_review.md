## 二、前提共识（诊断的地基）

1. synapse-rust 没有、也不需要外部采用率。它的两个真实身份是"学习工艺"和"HuLa 后端"。

## 三、逐条回答你最初的 5 个问题

### 1. 核心问题是什么？是"高性能 Matrix 服务器"还是别的？

**别的。** 不是"高性能 Matrix 服务器"——那是你对外的包装，也是让你痛苦的根源。真实核心是两件事叠在一起：
- **掌控**：HuLa 要一个你能 100% 改动、不受上游路线图约束的后端。
- **工艺/学习**：你想把一个完整 Matrix 服务器亲手实现一遍，深度学 Rust。

这两件事都成立、都有价值。问题只在于：你一直拿第三件不存在的事（"做成被广泛采用的服务器产品"）的尺子去量前两件，所以每次量都难看。

### 2. 已实现功能是否符合 Matrix 生态最迫切需求？

**这是个错位问题。** 你没有外部生态目标，"生态最迫切需求"对你不适用。换成能用的尺子：
- 对**学习轨**：13 个 MSC、E2EE、联邦全实现一遍，是极好的练习，越多越值。这里没有"沉没成本"，只有"练到没练到"。
- 对**交付轨（HuLa）**：唯一标准是 HuLa 客户端会不会调这个面。会调→留且必须稳；不会调→负重，冻结。

### 3. 私密扩展是否符合 Matrix 主流？是否该独立 fork？

- **不符合主流，也不该符合。** `com.hula.privacy`、阅后即焚、防截屏都是产品层 UX，不是协议层能力。用私有命名空间是对的，**不要**试图上游，会浪费时间且被拒。
- **诚实标注（重要）**：这两个是"软保证"，文档里必须写清楚，别误导用户当成硬保证——
  - 防截屏：服务端只能下发一个状态事件信号，真正拦截靠客户端（Android `FLAG_SECURE`）。信号能被改过的客户端无视。这是 UX 剧场，不是安全边界。
  - 阅后即焚：基于"收到 `m.read` 回执后服务端软删除"，是基于客户端诚实上报的信任模型，不是密码学保证。截屏、改过的客户端、抓包都能绕过。
- **是否独立 fork**：在两轨方案里，交付轨本来就是 HuLa 私有代码，不需要额外开源 fork。这些扩展留在交付轨即可。

### 4. 对照 element-hq/synapse v1.153.0，最影响采用率的 top 3 缺口？

"采用率"这个词请删掉——你没有采用目标。换成"最影响 HuLa 自己用得稳不稳"的 top 3：

1. **联邦（federation）** —— 如果 HuLa 是单 homeserver 部署（大概率是），跨服联邦是**最大的沉没成本**，不是缺口。反过来如果你确实要多服互通，那它的成熟度（房间版本、状态解析、PDU/EDU 边角）才是真缺口。先回答"HuLa 到底跨不跨服"。
2. **E2EE 边缘可靠性** —— cross-signing、device 验证、密钥备份的**恢复路径**。IM 用户真正会踩的是"换设备后历史消息解不开"。这块比多实现一个 MSC 重要得多。
3. **Spec 一致性（sytest / complement 通过率）** —— 决定 matrix-js-sdk fork 各种边角调用会不会突然 404 或行为不一致。这是 HuLa 稳定性的隐形地基，值得建一个可量化的通过率看板。

### 5. 哪些功能是沉没成本，应该砍掉以聚焦？

沉没成本 = 交付轨里 HuLa 不调用的东西。**先做一件事：抓一份 HuLa 客户端实际请求的路由清单（matrix-js-sdk fork 调用面 + 网络抓包），这是唯一权威的裁剪依据。** 在拿到清单前，以下是高度疑似沉没成本，按嫌疑排序：

| 候选 | 裁剪条件 | 处置（交付轨） |
|------|---------|--------------|
| 外部跨服联邦 | HuLa 单服部署 | 冻结/feature-gate 关掉，别删代码（留给学习轨） |
| SAML / CAS SSO | HuLa 只用密码 + OIDC | 冻结，只保留实际登录方式 |
| HuLa 不调用的 MSC | 对着调用清单逐个核 | 标 `#[cfg(feature="learning")]` 类隔离 |
| Elasticsearch 搜索 | PG FTS 够用 | 默认关，只留 PG 路径 |
| worker/replication 集群化 | 单机部署 | 冻结，别投维护精力 |

**注意用词是"冻结/隔离"不是"删"** —— 因为学习轨要留着它们。这正是两轨方案的价值：同一份代码，两把尺子，不打架。

## 四、选定路径：C —— 显式劈成两轨

你拍的板。把 synapse-rust 的身份**显式**分成两条轨，各自单独定义成功标准，从此不再问"对标上游"这种错位问题。

### 轨①：工艺/学习轨
- **成功标准**：你学到了、实现得优雅、你自己满意。就这样。
- **无采用压力、无路线图、无"对标上游"焦虑。** 13 个 MSC、完整联邦、SSO 三件套全部合法留存，它们是你的练习场。
- 最近这些 god-file 拆分、依赖注入重构（application_service、sliding_sync、storage 各种拆分）就是这条轨的正当产出——工艺打磨，不需要商业理由。

### 轨②：交付轨（HuLa 后端）
- **成功标准**：HuLa 能稳定、可靠地在生产环境跑起来，换设备不丢历史、私密功能行为可预期。
- **裁剪原则**：只为 HuLa 调用的面负责。不调用的一律 feature-gate 冻结（不是删）。
- **可靠性优先于功能数**：E2EE 恢复路径 + spec 一致性通过率 > 再实现一个 MSC。

### 两轨如何共存于一份代码
- 用 Cargo feature flag 把"HuLa 交付面"和"学习/完整面"分开（例如 `--features hula-minimal` vs `--features full`）。
- 交付部署只编交付面，攻击面和维护面都缩小；学习时编完整面，随便玩。
- 这样你既不用删掉心血（学习轨保住），又能给 HuLa 一个精简可靠的后端（交付轨聚焦）。
